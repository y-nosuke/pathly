# GPS軌跡の補正（スムージング）設計

## 概要

GPSの生データはノイズで「経路がぐちゃぐちゃ」になり、距離も実際より膨らむ。これを補正して、表示される軌跡と移動距離を実際に近づける。

> 対応する要望: [requirements.md](../requirements.md) の「記録した経路をきれいに見たい／正確な距離を知りたい」。ロードマップ Phase 2「GPSノイズ除去（位置補正）」。

## 設計方針

### 1. 非破壊 ＋ 補正後を保存（記録中に計算）

原データ（`gps_points`）は一切変更しない。補正後の点列は**別テーブル `smoothed_points` に保存**する。補正の計算は**記録中に増分で行い**、確定した点から順に保存する（従来の「読むたびに毎回計算」からの変更。詳細は後述「記録中の補正と保存」）。

- **記録中にライブ表示できる**: 補正後の軌跡・距離をその場で描ける。
- **一覧が軽い**: 表示のたびに全点を再計算しない。
- **削除しない方針に合致**: 生データは無改変。補正はあくまで別データとして併存させる（位置情報は個人に委ね自動改変しない、というプロジェクト方針と揃う）。
- **後から直せる**: しきい値やアルゴリズムを直したら、詳細画面から**再補正して上書き**できる（後述「再補正」）。記録中に計算・保存できなかったときの保険も兼ねる。

トレードオフ: 記録中に端末で計算するぶん電池・CPUを多少使う（**増分計算で「毎回全体」を避けて**抑える）。また補正後を保存するぶんスキーマ変更（Room v2→v3）が要る。

### 2. accuracy を「足切り」ではなく「重み」に

各GPS点には精度 `accuracy`（メートル）が付く。これを**ハードなしきい値**で切るのではなく、**重み**として使う。精度が悪い点ほど軽く扱うことで、マジックナンバー（精度上限・最小移動など）を減らし、「感覚で決める」要素を小さくする。

## アルゴリズム

`TrackSmoother.smooth(points, params)`（`domain/model/TrackSmoother.kt`）。2段階。

### Step 1: ジャンプ除外

直前の採用点との速度が `maxSpeedMps` を超える点を、明らかな外れ値として除外する。

```
速度 = 直前点との距離(m) / 経過秒
速度 > maxSpeedMps → その点を捨てる
```

しきい値は**乗り物を通し、GPSグリッチだけ落とす**高さにする。低すぎる（例: 55 m/s ≒ 200km/h）と新幹線（約83 m/s）が丸ごと除外され、混在トラック（徒歩→新幹線→徒歩）で中間区間が欠落し距離を過小評価する。既定は **150 m/s（≒540km/h）**：新幹線・特急は通し、1サンプルで数km飛ぶグリッチ（数百 m/s）は落とす。取りこぼしたノイズは Step 2 の精度重みが軽く扱う。区別の本質は「**速いが継続（乗り物）／1点だけ飛んで戻る（グリッチ）**」で、より頑健なスパイク検出は将来課題（「今後の拡張」参照）。

### Step 2: 精度重み付き移動平均

各点を、前後 `window` 点の**加重平均**に置き換える。重みは精度の二乗の逆数。

```
weight_j = 1 / (accuracy_j² + 1)
補正後の位置 = Σ(weight_j × 位置_j) / Σ(weight_j)   （窓の中の点 j について）
```

精度が良い（`accuracy` が小さい）点ほど強く効くので、窓の中に悪い点が混ざっても軽く扱われる。**始点・終点はずらさない**（端点は平均しない）。

## 記録中の補正と保存

補正は**記録中に増分で計算**し、確定した点を `smoothed_points` に保存する。処理は `LocationTrackingService`（GPS取得と同じ場所）で行う。

### 増分計算とラグ

- **Step 1（ジャンプ除外）は直前の採用点だけ見る**ので、新しい点が来た時点でオンラインに判定できる（ラグなし）。
- **Step 2（平滑化）は前後 `window/2`（既定 half=2）点が要る**ので、最新の末尾 `half` 点は**まだ確定できない**。点 `i` は `i + half` が到着した時点で確定し、`smoothed_points` に保存する。
- **ライブ表示**: 確定済みの点は補正後で、末尾の未確定 `half` 点は生（または片側だけの平均）で描く。ラグは既定で2点ぶん（10秒間隔なら約20秒）。
- **記録終了時**: 末尾の未確定点を端点扱いで確定させる。結果は一括バッチ計算（`TrackSmoother.smooth`）と一致する。

### 保存先: `smoothed_points` テーブル

補正後は生点と**一対一にならない**ため、`gps_points` への列追加ではなく**別テーブル**に持つ：

- `removeJumps` がスキップした生点には補正後が存在しない（行が無い）。
- 再補正でしきい値を変えると、生き残る点が変わる。

→ 「行の有無」でスキップを自然に表現でき、再補正は**そのtrackの行を全DELETE→再INSERT**で済む。

| カラム            | 型          | 備考                                          |
| ----------------- | ----------- | --------------------------------------------- |
| `id`              | INTEGER PK  | AUTOINCREMENT                                 |
| `track_id`        | INTEGER     | FK → `gps_tracks`（CASCADE）。index あり       |
| `seq`             | INTEGER     | track内の順序                                 |
| `latitude`        | REAL        | 補正後                                        |
| `longitude`       | REAL        | 補正後                                        |
| `source_point_id` | INTEGER?    | 由来の `gps_points.id`（任意・トレース用）     |
| `created_at`      | INTEGER     |                                               |

Room は v2→v3。破壊的フォールバックは無効なので `DatabaseMigrations` に正式なマイグレーションを追加する。`GpsTrack.smoothedPoints` は「読むたびに計算」から「保存済み `smoothed_points` を読む（無ければ生）」へ切り替える。

### 再補正（詳細画面）

詳細画面に「**軌跡を再補正**」導線を置く。押すと対象trackの `smoothed_points` を全消去し、生 `gps_points` から再計算して保存し直す。

- アルゴリズム／しきい値を直したあと、**過去の記録に反映**できる。
- 記録中に計算・保存できなかったときの**保険**。
- デバッグ版の調整UI（生↔補正の重ね表示・スライダー）と併存する。

## パラメータ（SmoothingParams）

| パラメータ    | 意味                          | 既定 | 大きくすると                   |
| ------------- | ----------------------------- | ---- | ------------------------------ |
| `maxSpeedMps` | ジャンプ除外の速度上限（m/s） | 150  | 除外が緩くなる（外れ値が残る）。新幹線を通すため高め |
| `window`      | 平滑窓（ならす点数・奇数）    | 5    | 滑らかになるが実際の角も丸まる |

既定値は `TrackSmoother` の定数（`MAX_SPEED_MPS` / `SMOOTHING_WINDOW`）。

### 平滑窓（window）の考え方

「1点をその前後何点と一緒にならすか」の範囲（奇数）。

- **小さい（3）**: 生に忠実。ノイズは少し残るが**実際の曲がり角を保つ**。
- **大きい（9〜）**: とても滑らかだが、**本物の角まで丸めて近道**してしまう。
- **1**: ならさない（＝補正なし）。

窓は**GPS間隔と連動**する。間隔を長くする（点が減る）と、同じ窓でも空間的に広い範囲をならすため強く効く。間隔を変えたら窓も見直す。

## 距離・指標

- **距離**: `TrackSmoother.totalDistanceMeters(points)`。`GpsTrack.totalDistanceMeters` は**補正後の点列**で計算するため、ノイズによる膨張が抑えられる。
- **ジャギーさ**: `TrackSmoother.totalTurningDegrees(points)`。連続区間の進行方向の変化量の合計（度）。ノイズでギザギザだと大きく、補正で下がる。「線の暴れ具合」を数値で比較する指標。

## 調整方法（客観化）

「最適」は真の経路が無い以上多少の判断が残るが、次で客観化する。

1. **調整ツール**（デバッグビルドのみ）: 詳細画面の「調整」トグルで、
   - 生データ（灰）と補正後（橙）を重ねて表示
   - スライダー（速度上限・平滑窓）で地図が即更新
   - 「距離: 生→補正」「曲がり角合計: 生→補正」を数値表示

   実装: `presentation/history/TrackDetailScreen.kt` の `tuningMode`（`BuildConfig.DEBUG` 限定）。

2. **指標の「ひざ」を見る**: 窓を上げると曲がり角合計が下がる。**下げ止まる手前**が「ノイズは取れて実形は残る」塩梅。
3. **正解経路でキャリブレーション（最も客観的）**: 距離が分かる経路を録り、**補正後距離が実測に一致**するよう合わせる。

## GPS間隔との連動

記録間隔は設定で変更可（`data/settings/SettingsRepository.kt`、既定10秒／5・10・30・60秒）。間隔は点密度を決め、補正の効き方（特に窓）に影響する。

## 決めた値の反映

調整ツールで良い値が固まったら、`TrackSmoother` の定数（`MAX_SPEED_MPS` / `SMOOTHING_WINDOW`）に反映する。通常モードの補正（`GpsTrack.smoothedPoints`）は既定パラメータ＝これらの定数を使うため、定数を更新すれば全体に効く。

## 今後の拡張

補正の「はしご」（下ほど強力・重い）。今回は 1〜2段目（ジャンプ除外＋精度重み付き平滑化）まで。3段目以降は将来検討する。

- **カルマンフィルタ**: 位置＋速度を状態として予測→観測で補正する、GPSの定番後段処理。**記録中（リアルタイム）と相性が良く**、未来の点が要らないため平滑化のラグ（前述「増分計算とラグ」）が原理的に出ない。今の重み付き平均を置き換える候補。非線形には Extended/Unscented KF。
- **マップマッチング**: 軌跡を道路ネットワークに吸着させ「経路をきれいに」する効果が最大。**OpenStreetMap のデータ＋ OSS エンジン（OSRM の `match` / Valhalla の Meili / GraphHopper など）**で実現できる（Python では Leuven.MapMatching / mappymatch 等）。QGIS 単体はマッチャーではなく、可視化・データ整備の土台という位置づけ。地図データか外部APIが要り重い・コスト増のため、見た目の劇的改善が欲しくなった段階で検討。
- **補正後の距離・点数をトラック行にも保存**して履歴一覧をさらに軽量化。
- 立ち寄り検出との連携（記録中の立ち寄り検出は別途仕様検討）。
