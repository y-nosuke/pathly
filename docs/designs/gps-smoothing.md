# GPS軌跡の補正（スムージング）設計

## 概要

GPSの生データはノイズで「経路がぐちゃぐちゃ」になり、距離も実際より膨らむ。これを補正して、表示される軌跡と移動距離を実際に近づける。

> 対応する要望: [requirements.md](../requirements.md) の「記録した経路をきれいに見たい／正確な距離を知りたい」。ロードマップ Phase 2「GPSノイズ除去（位置補正）」。

## 設計方針

### 1. 非破壊（原データは保持し、補正は毎回計算）

原データ（`gps_points`）は一切変更しない。表示・距離計算に使う補正後の点列は**読み込み時に計算**する。

- **調整が即反映**: しきい値を変えると過去の記録にもすぐ反映され、外出せずにチューニングできる。
- **マイグレーション不要**: スキーマを変えない。
- **削除しない方針に合致**: 位置情報は個人に委ね自動改変しない、というプロジェクト方針と揃う。

トレードオフ（読むたびに計算）は、点数が現実的な規模（10秒間隔で1時間＝約360点）では無視できる。将来、しきい値が固まり件数が増えたら、補正後の距離・点数をトラックに保存して一覧を軽量化する（[roadmap.md](../roadmap.md) 参照）。

### 2. accuracy を「足切り」ではなく「重み」に

各GPS点には精度 `accuracy`（メートル）が付く。これを**ハードなしきい値**で切るのではなく、**重み**として使う。精度が悪い点ほど軽く扱うことで、マジックナンバー（精度上限・最小移動など）を減らし、「感覚で決める」要素を小さくする。

## アルゴリズム

`TrackSmoother.smooth(points, params)`（`domain/model/TrackSmoother.kt`）。2段階。

### Step 1: ジャンプ除外

直前の採用点との速度が `maxSpeedMps` を超える点を、明らかな外れ値として除外する。

```
速度 = 直前点との距離(m) / 経過秒
速度 > maxSpeedMps → その点を捨てる
```

### Step 2: 精度重み付き移動平均

各点を、前後 `window` 点の**加重平均**に置き換える。重みは精度の二乗の逆数。

```
weight_j = 1 / (accuracy_j² + 1)
補正後の位置 = Σ(weight_j × 位置_j) / Σ(weight_j)   （窓の中の点 j について）
```

精度が良い（`accuracy` が小さい）点ほど強く効くので、窓の中に悪い点が混ざっても軽く扱われる。**始点・終点はずらさない**（端点は平均しない）。

## パラメータ（SmoothingParams）

| パラメータ    | 意味                          | 既定 | 大きくすると                   |
| ------------- | ----------------------------- | ---- | ------------------------------ |
| `maxSpeedMps` | ジャンプ除外の速度上限（m/s） | 55   | 除外が緩くなる（外れ値が残る） |
| `window`      | 平滑窓（ならす点数・奇数）    | 5    | 滑らかになるが実際の角も丸まる |

既定値は `TrackSmoother` の定数（`MAX_SPEED_MPS` / `SMOOTHING_WINDOW`）。

### 平滑窓（window）の考え方

「1点をその前後何点と一緒にならすか」の範囲（奇数）。

- **小さい（3）**: 生に忠実。ノイズは少し残るが**実際の曲がり角を保つ**。
- **大きい（9〜）**: とても滑らかだが、**本物の角まで丸めて近道**してしまう。
- **1**: ならさない（＝補正なし）。

窓は**GPS間隔と連動**する。間隔を長くする（点が減る）と、同じ窓でも空間的に広い範囲をならすため強く効く。間隔を変えたら窓も見直す。

## 距離・指標

- **距離**: `TrackSmoother.totalDistanceMeters(points)`。`GpsTrack.totalDistanceMeters` は**補正後の点列**で計算するため、ノイズによる膨張が抑えられる。
- **ジャギーさ**: `TrackSmoother.totalTurningDegrees(points)`。連続区間の進行方向の変化量の合計（度）。ノイズでギザギザだと大きく、補正で下がる。「線の暴れ具合」を数値で比較する指標。

## 調整方法（客観化）

「最適」は真の経路が無い以上多少の判断が残るが、次で客観化する。

1. **調整ツール**（デバッグビルドのみ）: 詳細画面の「調整」トグルで、
   - 生データ（灰）と補正後（橙）を重ねて表示
   - スライダー（速度上限・平滑窓）で地図が即更新
   - 「距離: 生→補正」「曲がり角合計: 生→補正」を数値表示

   実装: `presentation/history/TrackDetailScreen.kt` の `tuningMode`（`BuildConfig.DEBUG` 限定）。

2. **指標の「ひざ」を見る**: 窓を上げると曲がり角合計が下がる。**下げ止まる手前**が「ノイズは取れて実形は残る」塩梅。
3. **正解経路でキャリブレーション（最も客観的）**: 距離が分かる経路を録り、**補正後距離が実測に一致**するよう合わせる。

## GPS間隔との連動

記録間隔は設定で変更可（`data/settings/SettingsRepository.kt`、既定10秒／5・10・30・60秒）。間隔は点密度を決め、補正の効き方（特に窓）に影響する。

## 決めた値の反映

調整ツールで良い値が固まったら、`TrackSmoother` の定数（`MAX_SPEED_MPS` / `SMOOTHING_WINDOW`）に反映する。通常モードの補正（`GpsTrack.smoothedPoints`）は既定パラメータ＝これらの定数を使うため、定数を更新すれば全体に効く。

## 今後の拡張

- より高度な補正（カルマンフィルタ、立ち寄り検出との連携、地図マッチング）。
